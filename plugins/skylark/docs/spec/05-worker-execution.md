# Layer 5 — Worker Execution

## 1. Purpose

Worker execution is the layer that writes code. It receives a task specification and an expert prompt from upstream layers, provisions an isolated filesystem via git worktree, runs a single Claude Code CLI session in headless mode, and returns a structured JSON result. The worker is ephemeral — one task, one session, one exit. It never accumulates state across tasks.

This layer is the execution boundary between planning (Layers 1-4) and observation (Layers 6-7). Everything above it reasons about what to do; everything below it observes what happened. Layer 5 is the only layer that modifies application source code.

## 2. Components

| Component | Role |
|-----------|------|
| Claude Code CLI (`claude`) | Headless execution runtime. Runs in `--bare` mode with structured JSON output. |
| Git worktrees | Filesystem isolation. Each task gets its own worktree branched from the base branch. |
| Expert prompt (`.claude/CLAUDE.md`) | Vocabulary-routed identity and instructions written into the worktree by this layer, sourced from Layer 4. |
| Result writer | Parses CLI JSON output, writes `.skylark/results/TASK-NNN.json`, emits `WORKER_COMPLETE` event. |

### Claude Code CLI flags

| Flag | Purpose |
|------|---------|
| `--bare` | Skips all user config, OAuth, and MCP server initialization. Pure scripted mode with no side effects from the host environment. |
| `-p "prompt"` | Non-interactive single-prompt execution. The prompt is the full task instruction constructed from the task spec. |
| `--output-format json` | Structured output containing `session_id`, `cost_usd`, `duration_ms`, `num_turns`, `result`, and `is_error`. |
| `--max-turns N` | Hard upper bound on conversation turns. Prevents runaway sessions. Value set per risk level. |
| `--allowedTools [tools]` | Positive tool allowlist. When set, the worker can only use these tools. |
| `--disallowedTools [tools]` | Negative tool denylist. When set, these tools are removed from the worker's available set. |
| `--model` | Model override. Selects `sonnet` or `opus` based on risk level and dispatch context. |

### Git worktrees

Each task runs in a worktree created from the current base branch (typically `main`). The worktree provides:

- **Filesystem isolation.** The worker cannot modify files in the main working tree or other workers' worktrees.
- **Branch isolation.** Each worktree has its own branch. Commits do not touch the base branch until an explicit merge after a SHIP verdict.
- **Concurrent safety.** Multiple workers can execute in parallel without file conflicts (each has its own directory tree).

### Expert prompt installation

The expert prompt is generated by Layer 4 and written to `.skylark/experts/TASK-NNN.md` in the main tree. Layer 5 reads this file and writes it as `.claude/CLAUDE.md` inside the worktree. This ensures the Claude Code CLI loads it as the primary instruction set for the session.

The expert prompt contains:
- Identity statement (vocabulary-routed to the task's domain)
- Domain vocabulary clusters (15-30 terms)
- Anti-patterns (5-10 failure modes specific to the task)
- Operational guidance (error handling, concurrency model, edge cases)
- Testing expectations (framework, edge cases, performance targets)
- Deliverables (concrete files to create or modify)

## 3. Inputs

### Initial dispatch

Received from Layer 2 (Orchestrator) via `DISPATCH_WORKER` command:

```yaml
dispatch:
  task_id: number
  task_spec:
    title: string
    description: string
    details: string               # full implementation instructions
    acceptanceCriteria: string
    relevantFiles: [string]
    testStrategy: string
  expert_prompt_path: string      # .skylark/experts/TASK-NNN.md (in main tree)
  worktree_branch: string         # e.g. "task/TASK-042-fts5-search"
  max_turns: number               # default 20, configurable by risk
  model: sonnet | opus            # sonnet for standard, opus for critical
  risk: trivial | standard | elevated | critical
```

| Field | Required | Description |
|-------|----------|-------------|
| `task_id` | yes | Numeric task identifier from Layer 3 (Taskmaster). |
| `task_spec` | yes | Full task specification. All fields are passed inline — the worker never reads the task file. |
| `expert_prompt_path` | yes | Path to the expert prompt file in the main tree. Read by Layer 5 and installed as `.claude/CLAUDE.md`. |
| `worktree_branch` | yes | Branch name for the worktree. Convention: `task/TASK-NNN-<slug>`. |
| `max_turns` | yes | Hard turn limit for the CLI session. See Section 11 for defaults by risk. |
| `model` | yes | Model to use for the CLI session. See Section 8 for selection rules. |
| `risk` | yes | Risk level from triage. Drives model selection, tool scoping, turn limits, and timeout. |

### Re-dispatch (after REVISE verdict)

Received from Layer 2 when Layer 4 returns a REVISE verdict:

```yaml
re_dispatch:
  task_id: number
  worktree_path: string           # existing worktree from prior run
  findings: [{ severity, description, file, line }]  # from Layer 4 review
  round: number                   # current review round (1-indexed)
```

| Field | Required | Description |
|-------|----------|-------------|
| `task_id` | yes | Same task identifier as the original dispatch. |
| `worktree_path` | yes | Path to the existing worktree. The worker re-enters it — no new worktree is created. |
| `findings` | yes | Structured review findings from Layer 4. Each finding has severity (`error`, `warning`, `info`), description, file path, and line number. |
| `round` | yes | Current review round. The orchestrator enforces a maximum of 2 rounds before escalation. |

On re-dispatch, the worker:
1. Changes directory to the existing worktree (no `git worktree add`).
2. Reads the existing `.claude/CLAUDE.md` (expert prompt is already installed).
3. Constructs a fix prompt from the findings (not the original task spec).
4. Executes the CLI session.
5. Follows the same output flow as initial dispatch.

## 4. Worker Lifecycle

### Step 1: Create worktree

```bash
git worktree add <worktree-path> -b <worktree-branch>
```

**Worktree path convention:** `.worktrees/TASK-NNN-<slug>` relative to the repository root. All worktrees live under a single `.worktrees/` directory for predictable cleanup.

**Branch naming:** `task/TASK-NNN-<slug>` where `<slug>` is derived from the task title (lowercase, hyphenated, truncated to 50 characters). If the task has an `external_ref`, include it: `task/TASK-NNN-eng-142-<slug>`.

**Base branch:** The worktree branches from the current base branch (typically `main`). If the task depends on another task's branch that has not yet been merged, the orchestrator must merge or rebase that dependency first.

### Step 2: Install expert prompt

1. Read the expert prompt from `expert_prompt_path` (e.g., `.skylark/experts/TASK-042.md`).
2. Create the `.claude/` directory in the worktree if it does not exist.
3. Write the expert prompt as `.claude/CLAUDE.md` in the worktree.

The expert prompt becomes the Claude Code CLI's primary instruction set. The `--bare` flag ensures no other CLAUDE.md files (user, project, or global) are loaded — the worker sees only the expert prompt.

### Step 3: Construct task prompt

Assemble the prompt passed to `-p`:

```
## Task: {task_spec.title}

{task_spec.details}

## Acceptance Criteria

{task_spec.acceptanceCriteria}

## Relevant Files

{task_spec.relevantFiles, one per line}

## Test Strategy

{task_spec.testStrategy}

## Instructions

1. Implement the task as specified.
2. Write tests following TDD — write failing tests first, then implement.
3. Verify all tests pass.
4. Commit your work with a message referencing task {task_id}.
5. Self-review your work for completeness, quality, and discipline.
6. Report your status as one of: DONE, DONE_WITH_CONCERNS, NEEDS_CONTEXT, BLOCKED.

If anything is unclear, report NEEDS_CONTEXT with a description of what you need.
If you cannot complete the task, report BLOCKED with a description of what is preventing progress.
```

For re-dispatch, the prompt is instead:

```
## Review Findings — Round {round}

The following issues were found in your prior implementation. Fix them.

{findings, formatted as a numbered list with severity, description, file, line}

## Instructions

1. Address every finding above.
2. Do not introduce new issues while fixing these.
3. Run tests after fixes.
4. Commit your fixes.
5. Report your status.
```

### Step 4: Execute CLI

```bash
cd <worktree-path>
claude --bare \
  -p "<prompt>" \
  --output-format json \
  --max-turns <max_turns> \
  --model <model> \
  --allowedTools <tools> \
  --disallowedTools <disallowed_tools>
```

The process runs synchronously. The orchestrator waits for it to exit.

**Environment:** The CLI runs in the worktree directory. The `--bare` flag ensures no MCP servers, OAuth tokens, or user config are loaded. The only configuration the worker sees is the `.claude/CLAUDE.md` expert prompt.

### Step 5: Parse result

The CLI exits and writes a JSON object to stdout. Parse it to extract:

```json
{
  "session_id": "string",
  "cost_usd": 0.0,
  "duration_ms": 0,
  "num_turns": 0,
  "result": "string (the worker's final text output)",
  "is_error": false
}
```

From the `result` text, extract the worker's self-reported status (`DONE`, `DONE_WITH_CONCERNS`, `NEEDS_CONTEXT`, `BLOCKED`) and any accompanying details (concerns, blocked reason, context request).

Determine `files_changed` by running `git diff --name-only HEAD` in the worktree (comparing against the branch point).

### Step 6: Write result artifact

Write the full result to `.skylark/results/TASK-NNN.json` in the main tree:

```json
{
  "task_id": 42,
  "status": "DONE",
  "session_id": "sess_abc123",
  "total_cost_usd": 0.12,
  "duration_ms": 45000,
  "num_turns": 8,
  "files_changed": ["src/search/fts5.ts", "src/search/fts5.test.ts"],
  "concerns": null,
  "blocked_reason": null,
  "needs_context": null,
  "round": 1,
  "model": "sonnet",
  "risk": "standard",
  "timestamp": "2026-04-15T14:30:00Z"
}
```

### Step 7: Emit WORKER_COMPLETE event

Send the `WORKER_COMPLETE` event to Layer 2 (Orchestrator) with the `worker_result` payload. The orchestrator routes based on the status field.

### Step 8: Await verdict

The worktree is preserved until the orchestrator delivers a verdict:

| Verdict | Worktree action |
|---------|-----------------|
| SHIP | Merge worktree branch to base branch. Remove worktree via `git worktree remove`. Delete branch. |
| REVISE | Preserve worktree. Re-dispatch worker into it (Step 3 of re-dispatch). |
| RETHINK | Remove worktree via `git worktree remove`. Delete branch. Task returns to planning. |
| NEEDS_CONTEXT (re-routed) | Preserve worktree. Orchestrator enriches context and re-dispatches. |
| BLOCKED (escalated) | Preserve worktree. User intervention required. |

## 5. Outputs

### WORKER_COMPLETE event

Emitted to Layer 2 (Orchestrator):

```yaml
worker_result:
  task_id: number
  status: DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED
  session_id: string              # Claude Code session ID for audit trail
  total_cost_usd: number          # Total API cost for this session
  duration_ms: number             # Wall-clock time from CLI start to exit
  num_turns: number               # Conversation turns consumed
  files_changed: [string]         # Paths relative to worktree root
  concerns: string | null         # Set when status is DONE_WITH_CONCERNS
  blocked_reason: string | null   # Set when status is BLOCKED
  needs_context: string | null    # Set when status is NEEDS_CONTEXT
```

| Field | Type | Description |
|-------|------|-------------|
| `task_id` | number | Task identifier, echoed from dispatch input. |
| `status` | enum | Worker's self-reported completion status. |
| `session_id` | string | Claude Code session identifier. Used by Layer 6 for telemetry correlation. |
| `total_cost_usd` | number | Cumulative API cost for the session. Includes all turns, tool calls, and retries. |
| `duration_ms` | number | Wall-clock duration from CLI process start to exit. |
| `num_turns` | number | Number of conversation turns the session consumed. |
| `files_changed` | string[] | Files modified or created, relative to worktree root. Computed from `git diff`, not from the worker's self-report. |
| `concerns` | string or null | Free-text concerns when the worker completed but has doubts. Null unless status is `DONE_WITH_CONCERNS`. |
| `blocked_reason` | string or null | Free-text description of what is blocking progress. Null unless status is `BLOCKED`. |
| `needs_context` | string or null | Free-text description of what context is missing. Null unless status is `NEEDS_CONTEXT`. |

### Persisted artifact

Written to `.skylark/results/TASK-NNN.json` in the main tree. Contains the full `worker_result` plus metadata fields (`round`, `model`, `risk`, `timestamp`). Consumed by:

- Layer 6 (telemetry bridge) for cost and duration dashboards
- Layer 4 (review) for understanding what the worker attempted
- Audit trail for post-hoc analysis

### Taskmaster status update

On dispatch start, the worker updates Taskmaster via MCP: `set_task_status(task_id, "in-progress")`. On completion, the orchestrator (not the worker) updates Taskmaster based on the verdict.

## 6. Downstream

The `WORKER_COMPLETE` event is consumed by the orchestrator (Layer 2), which routes based on the `status` field:

| Status | Orchestrator action |
|--------|---------------------|
| `DONE` | Route to Layer 4 for code quality review (standard+ risk). For trivial risk, route directly to Layer 3 for status update. |
| `DONE_WITH_CONCERNS` | Route to Layer 4 for review. Concerns are included in the review context so reviewers can evaluate them. |
| `NEEDS_CONTEXT` | Orchestrator attempts to enrich context (query Taskmaster for related task outputs, read predecessor results). Re-dispatches to Layer 5 without incrementing the review round counter. If context cannot be resolved, escalate to user. |
| `BLOCKED` | Orchestrator attempts re-dispatch with a more capable model (e.g., sonnet to opus). If already at opus, escalate to user. |

### Merge flow (on SHIP verdict)

After Layer 4 returns a SHIP verdict, the orchestrator:
1. Merges the worktree branch to the base branch.
2. Removes the worktree via `git worktree remove <path>`.
3. Deletes the task branch.
4. Updates Taskmaster: `set_task_status(task_id, "complete")`.

### Telemetry bridge

The `.skylark/results/TASK-NNN.json` artifact is read by the Layer 6 telemetry bridge, which posts cost, duration, and turn count to Langfuse as trace metadata. The `session_id` field enables correlation with Claude Code's internal telemetry.

## 7. Worktree Management

### Creation

```bash
git worktree add .worktrees/TASK-NNN-<slug> -b task/TASK-NNN-<slug>
```

Preconditions:
- The `.worktrees/` directory exists (create if not).
- No existing worktree occupies the target path.
- No existing branch has the target name.

If the branch already exists (from a prior failed run), remove the stale worktree and branch before creating:
```bash
git worktree remove .worktrees/TASK-NNN-<slug> --force 2>/dev/null
git branch -D task/TASK-NNN-<slug> 2>/dev/null
git worktree add .worktrees/TASK-NNN-<slug> -b task/TASK-NNN-<slug>
```

### Isolation guarantees

- **No shared mutable state.** Each worktree has its own working directory, index, and HEAD. Workers cannot see each other's uncommitted changes.
- **Shared object store.** All worktrees share the same `.git` object store. This means `git fetch` in any worktree updates objects for all, but branch refs are independent.
- **No cross-worktree tool access.** The CLI runs in the worktree directory with `--bare`, so tool calls (file reads, writes, shell commands) are scoped to the worktree's filesystem subtree. The worker cannot navigate to sibling worktrees or the main tree.

### Cleanup triggers

| Trigger | Action |
|---------|--------|
| SHIP verdict | `git worktree remove <path>`, `git branch -d <branch>` |
| RETHINK verdict | `git worktree remove <path>`, `git branch -D <branch>` |
| Stale worktree (no activity for configurable timeout) | `git worktree remove <path> --force`, `git branch -D <branch>` |
| Pipeline abort (user cancellation) | `git worktree remove <path> --force`, `git branch -D <branch>` |

Cleanup is idempotent. Running cleanup on an already-removed worktree is a no-op.

### Merge conflicts

When merging a worktree branch back to the base branch:

1. Attempt `git merge --no-ff task/TASK-NNN-<slug>` from the base branch.
2. If the merge succeeds, proceed with cleanup.
3. If the merge conflicts:
   a. Abort the merge: `git merge --abort`.
   b. Attempt rebase: `git rebase <base-branch>` in the worktree.
   c. If the rebase succeeds, re-attempt the merge.
   d. If the rebase conflicts, escalate to the user with the conflict file list.
   e. The worktree is preserved for manual resolution.

Merge conflicts are most likely when multiple tasks modify the same files. The orchestrator's task DAG (Layer 3) should order dependent tasks to minimize this, but conflicts can still occur with shared files (e.g., index files, configuration).

### Gitignore

Add `.worktrees/` to the repository's `.gitignore` to prevent worktree directories from being tracked.

## 8. Model Selection

| Risk | Initial implementation | Re-dispatch after REVISE | Rationale |
|------|----------------------|--------------------------|-----------|
| trivial | sonnet | n/a (no review) | Trivial tasks skip review. Sonnet is sufficient for mechanical changes. |
| standard | sonnet | sonnet | Standard tasks get Sonnet for both passes. Cost-efficient for bounded work. |
| elevated | sonnet | opus | Elevated tasks escalate to Opus on revision. If Sonnet could not get it right the first time with review feedback, a more capable model may be needed. |
| critical | opus | opus | Critical tasks use Opus from the start. Load-bearing code warrants the strongest model throughout. |

### Model override

The orchestrator can override the model selection when:
- A `BLOCKED` status triggers an automatic capability escalation (e.g., sonnet to opus).
- The user explicitly requests a model for a specific task.

Model overrides are logged in the result artifact for cost attribution.

## 9. Tool Scoping

Workers are sandboxed to the minimum tool set required for their task. Tool scoping uses `--allowedTools` and `--disallowedTools` CLI flags.

### Default tool set (all risk levels)

All workers have access to:
- `Read` — read files
- `Write` — write new files
- `Edit` — edit existing files
- `Bash` — run shell commands (build, test, lint)
- `Glob` — find files by pattern
- `Grep` — search file contents

### Denied tools (all risk levels)

All workers are denied:
- `WebSearch` — workers do not browse the internet
- `WebFetch` — workers do not fetch URLs
- `Skill` — workers do not invoke skills (they are the skill's execution target)
- `NotebookEdit` — not applicable to standard development tasks

### Risk-based scoping

| Risk | Additional restrictions | Rationale |
|------|------------------------|-----------|
| trivial | `--disallowedTools "Bash"` for pure-edit tasks (no build/test step) | Minimizes blast radius for trivial changes. Bash is re-enabled if the task spec includes a `testStrategy`. |
| standard | No additional restrictions. | Standard tool set is appropriate. |
| elevated | No additional restrictions. | Same as standard; the review layer provides the additional safety. |
| critical | No additional restrictions. | Opus + full review pipeline is the safety mechanism, not tool restriction. |

### MCP tool scoping

Because `--bare` disables all MCP servers, workers have no access to MCP tools (Taskmaster, Linear, etc.) by default. The orchestrator handles all MCP interactions on the worker's behalf. This is intentional — workers write code, they do not manage tasks or external systems.

## 10. Error Handling

### Max turns exceeded

When the CLI session reaches `--max-turns` without the worker reporting a status:

1. The CLI exits with the session's final state.
2. Layer 5 marks the result as `BLOCKED` with `blocked_reason: "max turns exceeded without completion"`.
3. The orchestrator evaluates:
   - If `num_turns` is at the limit and `files_changed` is non-empty, the worker made progress but ran out of turns. Re-dispatch with a higher `max_turns` (up to the absolute ceiling).
   - If `num_turns` is at the limit and `files_changed` is empty, the worker is stuck. Escalate to user.

### CLI crash or non-zero exit

If the `claude` process exits with a non-zero code or produces malformed JSON:

1. Log the exit code and stderr.
2. Retry once with the same parameters (transient failures: OOM, network blip).
3. If the retry also fails, mark as `BLOCKED` with `blocked_reason: "CLI process failed: {exit_code}"`.
4. Preserve the worktree for inspection.

### Worktree creation failure

If `git worktree add` fails:

| Failure | Recovery |
|---------|----------|
| Branch already exists | Remove stale worktree and branch, retry. |
| Path already exists but is not a worktree | Remove the directory, retry. |
| Locked worktree (`.git/worktrees/<name>/locked`) | Unlock via `git worktree unlock`, retry. |
| Disk space | Escalate to user. |

### Result parsing failure

If the worker's final output does not contain a recognizable status keyword (`DONE`, `DONE_WITH_CONCERNS`, `NEEDS_CONTEXT`, `BLOCKED`):

1. Check if `is_error` is `true` in the CLI JSON output. If so, treat as `BLOCKED`.
2. If `is_error` is `false` and `files_changed` is non-empty, treat as `DONE_WITH_CONCERNS` with `concerns: "Worker did not report explicit status. Files were changed — review required."`.
3. If `is_error` is `false` and `files_changed` is empty, treat as `BLOCKED` with `blocked_reason: "Worker produced no status and no changes."`.

### Timeout

Each worker has a wall-clock timeout (see Section 11). If the CLI process exceeds the timeout:

1. Send SIGTERM to the process.
2. Wait 10 seconds for graceful shutdown.
3. If still running, send SIGKILL.
4. Parse whatever output was produced (if any).
5. Mark as `BLOCKED` with `blocked_reason: "Worker exceeded wall-clock timeout ({timeout_ms}ms)"`.
6. Preserve the worktree — partial work may be salvageable on re-dispatch.

### Concurrent worker conflict

If two workers are dispatched to tasks that share files (despite DAG ordering intended to prevent this):

- Each worker is in its own worktree, so there is no runtime conflict.
- The conflict surfaces at merge time (see Section 7, Merge conflicts).
- The orchestrator serializes merges — only one branch merges at a time.

## 11. Configuration

### Turn limits by risk

| Risk | Default max turns | Absolute ceiling | Re-dispatch max turns |
|------|------------------|------------------|-----------------------|
| trivial | 10 | 15 | n/a |
| standard | 20 | 30 | 20 |
| elevated | 25 | 40 | 30 |
| critical | 30 | 50 | 40 |

`max_turns` in the dispatch input overrides the default but cannot exceed the absolute ceiling. The ceiling is a safety valve — if a worker needs more turns than the ceiling, the task is mis-scoped.

### Timeout thresholds

| Risk | Wall-clock timeout |
|------|-------------------|
| trivial | 3 minutes |
| standard | 10 minutes |
| elevated | 15 minutes |
| critical | 25 minutes |

Timeouts are generous because they are a safety net, not a performance target. Most workers complete well within these limits.

### Model defaults

| Risk | Default model |
|------|--------------|
| trivial | `claude-sonnet-4-20250514` |
| standard | `claude-sonnet-4-20250514` |
| elevated | `claude-sonnet-4-20250514` |
| critical | `claude-opus-4-20250514` |

Model identifiers are pinned to specific versions to ensure reproducibility. Update these when upgrading model versions.

### Worktree path convention

```
<repo-root>/.worktrees/TASK-<id>-<slug>/
```

Examples:
- `.worktrees/TASK-042-fts5-search/`
- `.worktrees/TASK-013-eng-142-auth-middleware/`

### Result artifact path

```
<repo-root>/.skylark/results/TASK-<id>.json
```

Overwritten on re-dispatch. Prior results are not versioned — the latest run's result is the canonical one. Historical results can be recovered from the Layer 6 telemetry store.

### Expert prompt source path

```
<repo-root>/.skylark/experts/TASK-<id>.md
```

Written by Layer 4 before dispatch. Read by Layer 5 during worktree setup. Not modified by the worker.

### Review round limit

Maximum 2 review rounds per task. After round 2, the orchestrator escalates to the user regardless of verdict. This limit is enforced by the orchestrator (Layer 2), not by the worker.

### Stale worktree cleanup

Worktrees with no activity (no new commits, no running CLI process) for longer than 24 hours are eligible for automated cleanup. The orchestrator can run periodic garbage collection:

```bash
git worktree list --porcelain | # parse for stale entries
git worktree remove <path> --force
git branch -D <branch>
```
