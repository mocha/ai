# Pre-flight sizing and decomposition for agentic coding pipelines

**No major AI coding agent ships a pre-execution complexity estimator, but the building blocks to construct one exist today.** The empirical evidence from SWE-Bench shows that three metrics—files modified, lines changed, and cross-cutting concern count—predict agent failure rates with striking precision, and these can be computed statically before a single token is spent. For decomposition, the field has converged on DAG-based task graphs with vertical slice boundaries, but the critical insight from 18 months of production use across Devin, Cursor, and Factory is that **decomposition quality matters more than decomposition granularity**—over-decomposition fails as badly as under-decomposition.

This report maps the concrete techniques, tools, scoring rubrics, and failure modes relevant to Skylark's triage/prepare stage, drawing from Anthropic's official documentation, competitor systems, academic research, and open-source frameworks as of April 2026.

---

## The empirical case for pre-flight complexity scoring

The strongest quantitative foundation comes from SWE-Bench Verified analysis, which provides hard numbers mapping task characteristics to agent success rates. Jatin Ganhotra's difficulty stratification of 500 SWE-Bench instances reveals that **lines of code changed is the single strongest predictor**, with an 11× increase from easy to hard tasks, far outpacing file count (2×) or code hunks (5×).

The data supports a concrete scoring rubric. Easy tasks (under 15 minutes for a human) average **1.03 files, 1.37 hunks, and 5.04 lines changed**, with agent success rates above 80%. Medium tasks average 1.28 files, 2.48 hunks, and 14.1 lines changed, dropping success to roughly 62%. Hard tasks jump to **2.0 files, 6.82 hunks, and 55.78 lines changed**, with success rates plunging below 25%. Scale AI's SWE-Bench Pro further confirmed this: when tasks require multi-file modifications spanning hundreds of lines, even GPT-5 and Claude Opus 4.1 score only about **23%**—down from over 70% on simpler tasks.

Multi-file edits represent a cliff, not a slope. Only **3.09% of easy tasks** require multi-file changes, but **55.56% of hard tasks** do. Of 25 hard multi-file issues in SWE-Bench Verified, only 9 were solved by any system. This suggests Skylark's triage stage should flag multi-file scope as a hard gate for decomposition.

The METR/Anthropic paper "Measuring AI Ability to Complete Long Tasks" introduces **16 "messiness" factors** scored per task—novelty, dynamic environments, resource constraints, interaction requirements—and finds that each additional messiness point reduces success rates by approximately **8.1 percentage points** after controlling for task length. This provides a second scoring axis orthogonal to size: a small but messy task may need decomposition as urgently as a large clean one.

## A practical pre-flight scoring rubric for Skylark

Synthesizing the empirical data with Moderne's Prethink metrics and the Task Entropy Framework, Skylark's triage stage can implement a composite pre-flight score using statically computable signals. The following metrics can all be extracted before execution via AST parsing, dependency analysis, and git history.

**File blast radius** counts files likely to be modified. Static analysis can identify this by parsing the task description against a dependency graph. Moderne's Prethink tool generates pre-computed CSV files inside repositories containing method-level cyclomatic complexity, cognitive complexity, max nesting depth, parameter count, and a composite debt score weighted as `0.4 × norm(cyclomaticComplexity) + 0.3 × norm(cognitiveComplexity) + 0.2 × norm(maxNestingDepth) + 0.1 × norm(halsteadEstimatedBugs)`. At the class level, **LCOM4** (Lack of Cohesion of Methods) directly indicates how many classes a target should be split into—LCOM4 of 3 means the class should be three classes. **CBO** (Coupling Between Objects) counts external references and predicts blast radius. **WMC** (Weighted Methods per Class) sums cyclomatic complexities; a WMC above 100 signals that the class is too complex for safe agent modification.

**Context budget estimation** should calculate: files to read × average file size in tokens, plus tool definition overhead (Claude Code can consume **55K+ tokens** on tool schemas alone before the conversation starts), plus a 20–30% conversation overhead buffer. If the total exceeds **60% of the context window** before starting, the task should be decomposed. Anthropic's own documentation confirms that performance degrades progressively—not just at the limit—and community best practice is to manually compact at 60% utilization for best quality. Databricks Mosaic research found model correctness starts dropping after just **32K tokens** of accumulated context due to the "lost in the middle" effect.

The Task Entropy Framework from earezki.com (March 2026) formalizes two additional signals. **Blast radius signal**: single-file changes route to fast/cheap models; multi-file changes route to expensive/capable models. **Reversibility signal**: easy-to-undo changes get lower scrutiny; irreversible changes (database migrations, API contract changes) require maximum intelligence. A meta-agent assessing these signals can dynamically route tasks.

The Agent Psychometrics paper (arXiv:2604.00594, April 2026) demonstrates that pre-execution complexity estimation from static task features is viable without running the agent at all. Using Item Response Theory augmented with 12 features extracted from issue descriptions, test cases, and repository state, the framework predicts task-level success probability for unseen benchmarks and unseen LLM-scaffold combinations. The code is available at github.com/dariakryvosheieva/agent-psychometrics. A complementary approach, ComplexityNet (arXiv:2312.11511), fine-tuned a classifier to predict task complexity with **79% accuracy** and reduced computational usage by 90% while maintaining 86.7% code generation accuracy.

For Skylark's YAML frontmatter, a practical T-shirt sizing scheme derived from all this evidence:

- **XS**: 1 file, under 10 lines changed, no cross-cutting concerns → success rate above 80%, low context risk
- **S**: 1 file, 10–30 lines, no cross-cutting concerns → 60–80% success, low context risk
- **M**: 1–2 files, 30–60 lines, minor cross-cutting → 40–60% success, medium context risk
- **L**: 2–4 files, 60–150 lines, cross-cutting concerns present → 20–40% success, high context risk, decomposition recommended
- **XL**: 4+ files, 150+ lines, major cross-cutting concerns → under 20% success, very high context risk, decomposition mandatory

## What Anthropic and competitor systems actually do about task sizing

**Anthropic does not publish a specific ideal task size** in tokens or lines of code. Their Claude Code best practices documentation at code.claude.com/docs/en/best-practices frames the constraint behaviorally: "Most best practices are based on one constraint: Claude's context window fills up fast, and performance degrades as it fills." The recommended heuristic is that if you can describe the expected diff in one sentence, skip the plan; if you can't, use Plan Mode. After two failed corrections, `/clear` and start fresh—context polluted with failed approaches is worse than no context.

Claude Code's Task system (introduced in v2.1.16, January 2026) replaced flat todo lists with **DAG-based dependency tracking**. Tasks stored at `~/.claude/tasks/<task-list-id>/tasks.json` support explicit blocking relationships, and execution follows a graph traversal algorithm: scan for pending tasks where all blockers are completed, execute ready tasks in parallel via subagents, mark completed, check for newly unlocked tasks, repeat. Common patterns include linear chains, fan-out, fan-in, and diamond dependencies. The diamond pattern—where A fans out to B and C, which fan into D—is described as the most common in real projects. Setting `CLAUDE_CODE_TASK_LIST_ID` allows multiple Claude sessions to share a single task list, enabling multi-session coordination.

Claude Code Agent Teams (v2.1.32, February 2026, experimental) take this further with peer-to-peer messaging and shared task lists where teammates self-claim work. Unlike subagents that only report back to a parent, teammates coordinate directly. The tradeoff is significant: Agent Teams consume **3–4× the tokens** of a single session. Anthropic's April 10, 2026 blog post on multi-agent coordination patterns recommends starting with the orchestrator-subagent pattern and evolving only when it struggles.

Among competitor systems, **Devin** offers the most explicit sizing heuristic: "if you can do it in three hours, Devin can most likely do it." Their Managed Devins feature gives each subtask an isolated VM with its own terminal and browser, and a coordinator monitors ACU consumption and reads full agent trajectories to improve decomposition over time. **Cursor** lacks explicit complexity estimation but their research on scaling agents revealed a Planner/Worker architecture where planners recursively spawn sub-planners for specific codebase areas—tested at scale with hundreds of concurrent agents building million-line codebases. Their key lesson: "many improvements came from removing complexity rather than adding it." **Factory** uses robotics-inspired Model Predictive Control where plans continuously update based on environmental feedback, with recursive decomposition where "sub-issues decompose further if complexity thresholds are met." Their Terminal-Bench results demonstrate that **agent framework design produces greater improvements than model selection**—their Droid with Sonnet outperformed all other agents using Opus. **OpenAI Codex** supports configurable reasoning effort levels (low/medium/high/extra-high) per task, demonstrated 25-hour uninterrupted runs spanning roughly 13M tokens and 30K lines, and isolates each task in its own cloud sandbox.

No major system has built-in pre-execution complexity estimation. All rely on implicit signals—plan mode activation, reasoning effort selection, model routing—rather than explicit scoring.

## Decomposition methodologies that work in practice

The field has converged on three core decomposition patterns, each appropriate at different scales of complexity.

**DAG-based decomposition** represents subtasks as nodes with dependency edges, enabling parallel execution of independent work and enforcing prerequisites. Claude Code's Tasks system implements this natively. The ACONIC framework (arXiv:2510.07772) formalizes this by modeling tasks as constraint satisfaction problems, using the treewidth of constraint graphs to measure complexity, then applying minimal tree decomposition to partition into locally consistent subtasks—achieving **10–40 percentage point improvements** on benchmarks. For Skylark, this means the triage stage should output a task graph where each node specifies its file scope, inputs, outputs, and blocking relationships, stored as YAML that survives context resets.

**Vertical slice decomposition** is the single most important pattern for producing parallelizable subtasks. Each slice delivers end-to-end value through all layers—UI through API through service through database—rather than the horizontal decomposition (all database changes, then all service changes) that LLMs naturally tend to produce. HumanLayer's CRISPY methodology found that a **200-line design discussion provides 5× better leverage** than reviewing a 1,000-line plan when it captures the "design concept" that establishes shared context. The open-source Bokata Slicer (github.com/abrahamvallez/bokata-slicer-cc) automates vertical slicing for Claude Code using the "Hamburger Method," asking: "What would we ship if the deadline was tomorrow?" This directly addresses Skylark's need for subtasks suitable for parallel execution in separate context windows.

**Hierarchical task networks (HTN)** from classical AI planning categorize tasks as compound (needing further decomposition) or primitive (directly executable). The recommended depth is **2–3 levels for most tasks**, with each leaf-level subtask completing in 3–8 ReAct loop iterations. The TDAG framework (arXiv, Neural Networks 2025) extends this with dynamic decomposition and automatic sub-agent creation, addressing the critical problem of error propagation in static plans where an early failure cascades to all subsequent subtasks.

The RPI (Research-Plan-Implement) methodology, while community-developed rather than official Anthropic, has significant adoption and directly maps to Skylark's pipeline structure. Its three phases with validation gates—Research (spawning 3 parallel sub-agents for codebase location, analysis, and pattern discovery), Plan (interactive phased plan creation), and Implement (execution in isolated worktrees)—reduced one practitioner's fix-it commits from **47% to under 10%**. The evolution QRSPI splits this into 8 phases (Question → Research → Design → Structure → Plan → Worktree → Implement → PR), recognizing that the original 85+ instruction prompts exceeded LLM instruction-following limits.

## What makes a task atomic enough for a single agent session

A recent paper, "Scaling Coding Agents via Atomic Skills" (arXiv:2604.05013), identifies **five fundamental atomic skills**: code localization, code editing, unit-test generation, issue reproduction, and code review. These serve as "basis vectors" for complex software engineering tasks—composable, generalizable, and independently trainable. Joint reinforcement learning over these skills improved SWE-bench Verified scores from 0.507 to 0.585.

The practical criteria for a well-decomposed atomic task converge across sources. The task must **fit in one agent session** without exceeding context limits. It must have **unambiguous pass/fail criteria**—the agent must know what "done" looks like. If a feature requires **multiple context switches, it's too big**. Continue.dev's taxonomy classifies tasks as Type 1 (narrow/straightforward, one right answer), Type 2 (specific but needs context), and Type 3 (big/open-ended)—with the rule that Type 3 tasks should never be assigned directly to an AI but instead broken into Type 1 and 2 tasks first.

Task contracts formalize these criteria. The "Prompt Contracts" pattern uses an 8-section specification: objective, pre-conditions, invariants (what must NOT change), file scope, implementation contract, test-first spec, acceptance criteria using the 5C rubric (Countable, Constrained, Comparable, Checkable, Cheap), and an explicit anti-pattern list. GitHub's open-source Spec Kit implements a similar flow: Spec → Plan → Tasks → Implement → Review, where each task should be implementable and testable in isolation. The key insight from practitioners: "The most dangerous failure mode is the **completeness gap**—the contract doesn't cover something, so the agent fills it with assumptions."

## Failure modes that destroy decomposition quality

**Compounding error probability** is the mathematical foundation of decomposition risk. If an agent makes the right decision 95% of the time at each step, after 10 steps the probability of a fully correct outcome is **0.95^10 ≈ 60%**. After 20 steps, it drops to 36%. This means decomposition must minimize total steps, not just task size.

**Over-decomposition** is as dangerous as under-decomposition. Amazon Science's research warns that "excessive decomposition can increase complexity and coordination overhead to the point of diminishing returns" and risks "sacrificing the novelty and contextual richness that LLMs can provide by capturing hidden relationships within the complete context." Cursor's team discovered the same principle: their best results came from "removing complexity rather than adding it," including eliminating an "integrator" role that had become a bottleneck.

**Hidden dependency failures** occur when Agent A's changes invalidate Agent B's assumptions without any signal. Augment Code identifies this pattern explicitly: "If Agent A builds the API and Agent B builds the frontend calling that API, and Agent B needs Agent A's endpoint signatures while working, worktree isolation will not help." The coordinator must detect these dependencies and sequence dependent tasks rather than parallelizing them. The MAST taxonomy (UC Berkeley, March 2025) documents 14 failure modes across 1,642 annotated traces, including improper task decomposition where the planner creates unusable fragments, role confusion where planners start writing code, and incomplete verification where code is accepted merely because it compiles.

**Horizontal decomposition** is the anti-pattern LLMs naturally produce—all database changes, then all service layer, then all API, then all frontend. Nothing works until everything is complete, making testing impossible at each step. Vertical slicing directly counters this but requires explicit instruction.

**Specification drift** occurs when the agent's understanding diverges from original intent over iterations. Fixed with explicit goal restatements in each subtask contract and periodic alignment checks. **Context contamination** between subtasks happens when one agent's failed approaches pollute shared state. Anthropic's two-failure rule—`/clear` and restart after two failed corrections—applies at the subtask level too.

## Parallel execution requires worktree discipline and merge-time validation

Git worktrees have become the standard isolation primitive for parallel AI agent execution. Each agent gets its own worktree (branch plus working directory) sharing the underlying `.git` object store, preventing concurrent file overwrites, context contamination, and git lock contention. Tools like Worktrunk (worktrunk.dev), Superset (github.com/superset-sh/superset, supporting 10+ parallel agents), and ComposioHQ's agent-orchestrator all build on this pattern.

Practical gotchas that Skylark's implementation should handle: **port conflicts** where every dev server defaults to the same ports—solved with `SERVICE_PORT = BASE_PORT + (WORKTREE_INDEX * 10) + SERVICE_OFFSET`; **dependencies not carrying over** to new worktrees (node_modules, virtual environments); and critically, **no cross-worktree conflict warning**—Git has no mechanism to warn when two worktrees modify the same files on different branches. Block's open-source agent-task-queue (github.com/block/agent-task-queue) implements a FIFO queue to prevent concurrent expensive operations like builds and test runs from thrashing shared resources.

Practitioners report that **5 parallel agents** is a standard workflow with well-chosen independent tasks producing fewer than 3 merge conflicts per day over weeks of operation. The atomic workflow recommendation is "**parallel research, atomic implementation**"—research tasks are inherently parallelizable (one agent finding authentication patterns doesn't block another analyzing database schemas), but implementation should be sequential unless file scopes are provably disjoint.

## Concrete tools and frameworks for Skylark's triage stage

For pre-flight analysis, **Moderne Prethink** (docs.moderne.io/user-documentation/agent-tools/prethink/) is the most production-ready tool, generating structured context files in `.moderne/context/` with method-level, class-level, and package-level quality metrics plus test gap analysis, dependency inventories, and coupling-cohesion quadrant classification. It auto-updates CLAUDE.md to reference generated context. The **Agentic Engineering Framework** (dev.to/irrindar) maintains YAML component cards tracking `depends_on` and `depended_by` relationships, with `fw fabric blast-radius HEAD` computing dependency edges from commits—directly implementing the "task blast radius" concept.

For decomposition orchestration, **safethecode/orc** (github.com/safethecode/orc) is the most algorithmically sophisticated open-source option, with a dedicated decomposer, router, scheduler, worker pool, supervisor, and doom-loop detection in its core module. It includes codebase mapping, context compression, tree-sitter-based safety classification for shell commands, and optimization harnesses that run phased tournaments with parallel strategies. **ComposioHQ/agent-orchestrator** (github.com/ComposioHQ/agent-orchestrator) focuses on operational fleet management—each agent gets its own worktree, branch, and PR, with CI failures automatically routed back to agents for self-repair and reviewer comments forwarded for resolution.

For vertical slicing, **Bokata Slicer** (github.com/abrahamvallez/bokata-slicer-cc) integrates directly with Claude Code as slash commands. For spec-driven decomposition, **GitHub Spec Kit** and the **Prompt Contracts** pattern provide structured specification templates. For multi-agent coordination, **Cline Kanban** (cline.bot/kanban) offers agent-agnostic orchestration with dependency chains and automatic task chaining—it works with Claude Code, Codex, and other agents, not just Cline itself.

## Conclusion

Skylark's triage stage should implement a two-gate system. **Gate 1** computes a composite complexity score from statically available signals—file blast radius via dependency graph analysis, estimated lines changed from task description parsing, class-level WMC and CBO from pre-computed metrics (Moderne Prethink or equivalent), and a context budget estimate summing expected file reads, tool definitions, and conversation overhead against a 60% context window threshold. Tasks scoring above the threshold proceed to Gate 2. **Gate 2** performs DAG decomposition using vertical slice boundaries, producing subtask contracts with explicit file scopes, input/output specifications, invariant lists, and testable acceptance criteria. Each subtask is validated for independent executability—if it can't be tested in isolation, the decomposition is rejected and re-attempted with different slice boundaries.

The most underappreciated finding across all this research is that **the decomposition prompt matters more than the decomposition algorithm**. Factory's Terminal-Bench results show framework design outweighing model selection; Cursor's scaling research shows that removing structural complexity often outperforms adding it; and the compounding error probability math demonstrates that minimizing total agent steps matters more than minimizing individual step size. For Skylark, this means investing in the quality of task contracts—objective, pre-conditions, invariants, file scope, acceptance criteria—will yield better returns than investing in more sophisticated decomposition graph algorithms. The contract is the product; the graph is just the execution order.
