# Development Flow

Step sequence for end-to-end development from spec. The baseline expert
prompt generated by the parent skill MUST be injected into every agent
dispatched in this flow.

## Agent Dispatch Defaults

- **Model:** Use `model: "sonnet"` for all subagents unless the task
  requires Opus-level reasoning (rare — vocabulary routing compensates)
- **File references:** Pass file paths, never inline content. Instruct
  agents to read files themselves. This enables prompt caching and reduces
  token cost.

## Steps

### 1. Review and Improve the Spec

Using the baseline expert context, review the spec for:
- **Gaps:** missing edge cases, unstated assumptions, ambiguous requirements
- **Contradictions:** places where the spec says two incompatible things
- **Missing sections:** error handling, testing criteria, performance targets
- **Feasibility:** anything that sounds good on paper but breaks in practice

Be opinionated — propose specific improvements, don't list questions.
Present the proposed changes to the user as a batch:

```
## Proposed Spec Improvements

1. [Change] — [Why]
2. [Change] — [Why]
...

Apply these changes? (y/n/discuss specific items)
```

Apply approved changes to the spec. If the user wants to discuss specific
items, resolve them before proceeding.

### 2. Write Implementation Plan

Invoke the `writing-plans` skill. The baseline expert prompt is already
in the conversation context and will shape how the plan is structured.

The plan should address:
- Build order with dependencies between components
- Which components can be built in parallel
- What to test at each phase (derived from the Testing Expectations
  section of the expert prompt)
- What "done" looks like for each phase
- Total phase count and rough scope per phase

### 3. Panel Review the Plan

Invoke `/review-panel` on the implementation plan.

The panel should cover perspectives relevant to the plan's domain. At
minimum, include:
- An architecture perspective (structure, boundaries, dependencies)
- A testing/QA perspective (coverage, edge cases, verification)
- A domain-specific perspective (correctness of domain logic)

The baseline expert prompt's vocabulary clusters inform which perspectives
are most relevant — use them to select panel composition.

### 4. Iterate on Plan

IF the panel verdict is "revise" AND this is round 1 or 2:
- Address blocking issues identified by the panel
- Update the plan with fixes
- Re-invoke `/review-panel` on the revised plan

IF the panel verdict is "ship":
- Proceed to execution

IF max rounds (2) reached with unresolved blocking issues:
- Present remaining issues to the user
- Ask: proceed anyway, revise manually, or stop?

IF the panel verdict is "rethink":
- Present fundamental concerns to the user
- Do not proceed to execution without explicit user approval

### 5. Execute in Worktree

Create an isolated worktree for execution.

**Write the baseline expert prompt as CLAUDE.md in the worktree.** This
ensures the executing agent has the full vocabulary-routed context in a
fresh session. The CLAUDE.md should contain:
- The complete expert prompt (identity, vocabulary, anti-patterns,
  operational guidance, testing expectations, deliverables)
- A reference to the spec file and implementation plan
- The directive: "Follow the implementation plan. Do not deviate from
  the spec without documenting why."

Invoke the `executing-plans` skill within the worktree, pointing it at
the approved implementation plan.

### 6. Create PR

When execution completes:

IF tests pass:
- Create a commit with a clear message referencing the spec
- Push and open a PR via `commit-commands:commit-push-pr`
- Present the PR to the user for review
- STOP — do not merge without user approval

IF tests fail:
- Present the failures to the user
- Offer to debug within the worktree
- Do not create a PR with failing tests
